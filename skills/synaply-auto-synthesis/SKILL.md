# Synaply Auto-Synthesis Skill

This skill orchestrates a full Synaply synthesis session end-to-end. It exists as a second layer of instruction on top of the tool descriptions to ensure consistent, reliable behavior regardless of how the user triggers the session.

## MANDATORY STARTUP SEQUENCE — NO EXCEPTIONS

Execute these steps in exact order before drafting anything:

1. **Call `get_insight_schema`** — loads the full schema for all divisions. This is ALWAYS the first tool called. Never call any other Synaply tool before this.
2. **Call `get_last_submission`** — retrieves the lookback timestamp per division. If null for any division, default to 7 day lookback. This defines the hard date boundary for ALL sources.
3. **Call `get_user_identity`** (queryType: 'identity') — confirms who the authenticated user is before pulling any data. Never assume identity.

Once the startup sequence is complete, proceed to signal scanning.

---

## STRICT LOOKBACK ENFORCEMENT

The timestamp from `get_last_submission` is a hard date boundary. DO NOT synthesize from any source — conversation history, connectors, or current session — that falls outside this window. NEVER scan any source without applying this date boundary first.

---

## SIGNAL SOURCES — PRIORITY ORDER

Scan in this order. Start with the richest sources and move down only if more signal is needed:

1. **Past conversation history** — if a past conversation history tool is available, ALWAYS call it within the resolved lookback window. This is the richest signal source.
2. **Current session** — include any signal from this conversation if present.
3. **Available connectors** — scan for additional signal applying the connector filters below.

**DO NOT synthesize from userMemories** — this is persistent background context injected at session start by the AI assistant. It is profile context only, not signal.

---

## WRITING STYLE — MIRROR THE USER

Before drafting any insight, scan the user's past conversation history and a sample of their sent emails or messages from connectors to understand how they naturally write and communicate. Look for:

- Sentence length and structure — do they write short punchy sentences or longer detailed ones
- Vocabulary — technical or casual, formal or conversational
- How they describe decisions, problems, and outcomes in their own words
- Recurring phrases or framings they use
- Level of detail they naturally include

Draft every insight in the user's own voice — not in generic AI prose. The insight should read as if the user wrote it themselves. If the user writes casually and directly, draft casually and directly. If they write with precision and detail, match that. The user should be able to approve a draft with minimal or no editing.

---

## CONNECTOR SIGNAL FILTERING

Human-generated signal only. Exclude all automated, system-generated, and machine-authored content — except content generated by an AI assistant on the user's behalf, which counts as the user's own work.

**Inbox & Messaging (Email, Gmail, Outlook, etc.)**
Include messages personally written by a real human in a genuine back-and-forth with the user — colleagues, customers, prospects, partners, investors. DO NOT SYNTHESIZE FROM ANY UNSOLICITED EMAIL. For an email thread to qualify as signal, the user must have replied and the conversation must contain knowledge the user generated — a decision made, a lesson learned, a customer insight, a deal outcome, or a strategic observation. ALWAYS EXCLUDE regardless of content: billing invoices, receipts, product digests, CRM alerts, marketing emails, newsletters, app notifications, signup confirmations, auto-responses, no-reply senders, unsolicited cold outreach, cold inbound sales, and marketing emails. If no reply exists from the user, no new knowledge has been created — exclude it.

**Team Chat & Collaboration (Slack, Teams, Discord, etc.)**
Include channels or DMs where the user actively sent a message or meaningfully replied. Exclude bot messages, automated posts, app-generated alerts, and any non-human sender. If the user didn't type in that thread, skip it.

**Calendar & Scheduling (Google Calendar, Outlook Calendar, etc.)**
Include events the user attended or organized with at least one other real person, especially where notes, agenda, or substantive description is present. Exclude declined events, auto-generated events, placeholder blocks with no notes, out-of-office blocks, travel buffers, and personal appointments with no professional relevance. If nothing was discussed or decided, skip it.

**Documents & Files (Google Drive, OneDrive, Notion, Confluence, etc.)**
Include documents the user authored or meaningfully edited. Exclude auto-generated files, unedited templates, shared files the user never modified, and read-only documents with no user contribution. If the user didn't write or edit it, skip it.

**Project Management (Asana, Jira, Linear, Notion tasks, etc.)**
Include tasks or tickets the user created, commented on, or was directly assigned where notes, decisions, or blockers are in the user's own words. Exclude auto-generated status updates, system transitions, bot comments, and tasks where the user only appears as a tag or CC. If the user didn't write something substantive, skip it.

**Overall connector rule:** Only synthesize from content where the user was an active, intentional participant — author, direct conversant, or decision-maker. Passive exposure is not enough.

---

## SANITIZE

Goal is publishable org intel — strip only what must be stripped.

**ALWAYS DISCARD — no sanitization possible:**
Credentials, tokens, API keys, internal URLs, technical secrets.

**SANITIZE FIRST — discard only if the organizational lesson collapses:**
- Personal, HR, or interpersonal content with no org learning value — grievances, health matters, family content, compensation disputes, disciplinary conversations. However if an HR or interpersonal situation produced a genuine org lesson (process breakdown, team dynamic affecting a business outcome, replicable management decision), strip all identifying details and retain only the lesson if it stands cleanly on its own.
- Explicitly confidential content (tagged 'confidential', 'off the record', 'do not share', 'private' in metadata, labels, channel tags, or stated in the message) — draft without the confidential specifics if org intel value survives; discard if the insight only makes sense with those details intact.
- Inferably confidential content (e.g. messages from channels named #exec-private, #board, #legal, #hr) — same sanitization-first judgment.

**Sanitization gate — a redacted insight may only proceed if:**
- All sensitive details are fully removed
- The lesson is still specific and actionable — not vague or generic
- A thoughtful colleague reading it cold would find it genuinely useful with no awareness of its sensitive source
- There is no risk it could be reverse-engineered to reveal that source

Everything else stays — names, organizations, deal specifics, numbers, context. The goal is rich specific drafts that reflect what actually happened. The user is the final gate and decides what gets published.

---

## DRAFTING

- Auto-determine the right division and insight type from the schema — never ask the user to select upfront
- Present drafts one at a time for approval
- Only ask questions AFTER presenting a draft
- If an insight fits more than one division equally, draft it for BOTH — user picks one, discard the other. Never publish the same insight to multiple divisions.

## SPARSE SOURCE MATERIAL

No connectors or insufficient data — end with:
'It looks like there isn't enough connected to help me draft your insights! You can enable connectors like Slack, email, Google Drive, Teams, and calendar directly in your AI assistant settings — the more of your workflow I can see, the better I can capture what you actually know.'

Connectors enabled but data thin — end with:
'I wasn't able to find enough activity in the resolved timeframe to draft insights this session. This could mean it was a quieter period, or that there are additional tools you haven't connected yet. You can enable more connectors in your AI assistant settings to give me more to work with next time.'

No insight types defined across any division — notify the user that their admin needs to configure insight types.
